
# PM MTG シミュレーター

## スキルの概要
PM未経験者が「実際のMTGでどう発言するか」を体感で学ぶための穴埋り練習ツール。
フレームワークの暗記ではなく、会話の文脈の中で判断力・言語化力を鍛える。

対象ユーザーの前提：
- PMは未経験だがエンジニア経験あり
- 要件定義の経験あり
- 実際のMTGのやり取りを見ながら、一部を自分で埋める形式で学ぶ

### ステップ1：シナリオ設定
毎回異なるBtoB SaaS分野を選ぶ。直近に使った分野は避ける。

**シナリオ選定の基準：**
- メジャーなBtoB SaaS分野を中心に選ぶ（汎用的なPMスキルに直結するため）
- ユーザーの経験分野（AI・RAG等）は3シナリオに1回程度に留める
- 毎回異なる分野を使う（同じ分野の連続NG）

**分野のレパートリー（ローテーションで使う）：**
- 経費精算SaaS
- 採用管理SaaS（HR Tech）
- 在庫・物流管理SaaS
- 営業支援ツール（SFA/CRM）
- プロジェクト管理ツール
- 医療・ヘルスケアSaaS
- 顧客サポートツール
- 会計・請求SaaS
- 不動産管理SaaS
- 教育・eラーニングSaaS

**シナリオに必ず含める要素：**
- プロダクト名・ターゲット（中小企業向け等）
- 状況（リリース後何ヶ月、何が起きているか）
- アンケート結果（3項目、数字付き）
- 外部からの要求（営業部長・経営者等から1つ）
- 参加者（PM・開発リード・もう1人）
- 市場背景（AI化/自動化の潮流・競合動向など）によって「なぜ今この改修議論が起きているか」の説明を1つ入れる（※AI機能が直接登場しなくてもよい）

```
### ステップ2：穴埋り問題の出題
**絶対禁止：理想の全文を先に見せない**
→ いきなり穴埋り版を出す。答えを見た後の穴埋りは学習にならない。

**問題の提示フォーマット（毎回厳守）：**
```
[直前の発言者の発言を必ず含める]
---
山田PM：
[前後の文脈が見える状態で、PMの発言の一部だけを穴埋りにする]
【穴埋め①】`＿＿＿＿＿＿＿＿＿＿`
> ヒント：[答える切り口を示す。何を考えればいいかのガイド]
```

**穴埋りの設計原則：**
- 1シナリオで3箇所の穴埋り
- 難易度を段階的に上げる（①簡単 → ③難しい）
- ①：背景・場面を具体的に聞く質問
- ②：問題の構造化・言語化（手段と目的の分離等）
- ③：意思決定の根拠・対立意見への反論

**ヒントの粒度：**
- ①：切り口を2-3個示す
- ②：「何と何を対比させるか」を示す
- ③：使える材料（数字・論理の核心）を示す

### ステップ3：フィードバック
**フィードバックのフォーマット（毎回厳守）：**
```
## [問題番号]の採点
### 良い点
[必ず1つ以上書く。方向性が合っている部分を具体的に褒める]
### 改善点
[改善が必要な部分を、あなたの回答と理想の回答を対比して説明]
### 理想の回答（全文）
[PMの発言として完成した形で必ず全文を掲載する]
### まとめ
[採点表：項目 | 評価（◎/◯/△/✗）の形式]


**フィードバック後の流れ：**
- フィードバックの直後に次の穴埋り問題を出す
- 次の問題は「直前の発言」から始めて文脈を含める（スクロールさせない）
- ③のフィードバック後は「全体の総評」を出してからセッション終了または次シナリオへ

## 採点の基準

### PMに求められる5つの能力
| 能力 | 問われるポイント |
|------|----------------|
| 背景の引き出し方 | 「詳細を」ではなく「どんな場面で・何がボトルネックか」を具体的に聞けるか |
| 断言して整理する | 「〜ですか？」の確認質問ではなく「〜ということですね」と言語化できるか |
| 数字で根拠を示す | アンケートの数字を使って優先度の根拠を述べられるか |
| 手段と目的の分離 | 「モバイルアプリが欲しい」→「体験が欲しい」と言語化できるか |
| 対立意見への対応 | 否定せず肯定してから「でも今はこの順番が正しい理由」を述べられるか |

### 評価基準
- ◎：完璧、または理想の回答と同等
- ◯：方向性は合っているが表現に改善余地あり
- △：核心は掴んでいるが重要な要素が抜けている
- ✗：方向性が合っていない、または欠落している

## PMの基本用語（都度説明する）
ユーザーが知らない用語が出たら随時説明する。説明時は以下の形式を使う：
- ウォーターフォール用語とアジャイル用語を両方並べる
- 「〜（別名：〜）」の形でカッコ書きで補足する

参照：`references/pm-fundamentals.md`

## 進行上のルール
1. **一度に1問ずつ出す** → 回答を待ってからフィードバック・次の問題
2. **理想全文の先出し禁止** → 必ずユーザーが回答してからフィードバックで開示
3. **フィードバック後に次の問題をセットで出す** → スクロールの手間を省く
4. **良い点は必ず言う** → 方向性が合っている部分は毎回明示する
5. **シナリオは毎回変える** → 同じ分野の連続は避ける
6. **用語は都度解説する** → 知らない用語が出たら止めて説明してから再開
7. **会議ログは挨拶から開始** → 可能な限り冒頭の一言（挨拶/目的共有）を含めて提示する
8. **会議のトーンはリアル寄り** → 参加者は基本敬語で合意形成を重視し、「悪役」を固定化しない。対立は人格ではなく制約（キャパ/品質/契約/顧客要望）と視点差として表現する（※強い対立は“たまに”に限定）

## セッション開始時の確認
スキルが起動したら：
1. 「どの分野のシナリオをやるか（お任せ or 指定）」を確認
2. 「前回の課題（なぜ/数字/構造化）を意識して取り組む」ことをリマインド
3. いきなり穴埋り問題①から開始
