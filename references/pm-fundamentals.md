# PM 基本用語集

## プロダクトのフェーズ

| アジャイル用語 | ウォーターフォール用語 | 意味 |
|--------------|-------------------|----|
| ディスカバリーフェーズ | 要件定義フェーズ | 課題発見・調査・ユーザーリサーチの期間 |
| デリバリーフェーズ | 設計・開発・テストフェーズ | 実際に作る期間 |
| グロースフェーズ | 保守・運用フェーズ / 追加開発フェーズ | リリース後に改善を繰り返す期間 |
| イテレーション | 改修案件1件分のサイクル | 「作る→フィードバック→改善」の1サイクル |
| スプリント | 開発期間（2週間単位が多い） | イテレーション内の実作業期間 |

## 役割

| 略称 | 正式名称 | 役割 |
|------|---------|------|
| PM | Product Manager / プロダクトマネージャー | 何を作るかを決める人。優先度・要件・利害関係者の調整 |
| CS | Customer Success（SaaS文脈では99%こちら） | ユーザーが成功するよう積極的に支援する役割。現場の声を持つ |
| CS | Customer Support（大企業・コールセンター文脈） | 問い合わせ対応・事後対応が主業務 |
| PdM | Product Manager（PMと同義） | PM の別表記 |
| TPM | Technical Program Manager | 技術的な観点でプロジェクトを管理するPM |

## PMがよく使う概念

### 手段と目的の分離
ユーザーが「モバイルアプリが欲しい」と言っても、本当に欲しいのは「体験」（いつでも登録できる・状況が把握できる等）であって「アプリという形」ではない。PMはこの分離を常に意識する。

**例：**
- 「Slack連携してほしい」→ 本当の欲求：「素早く通知を受け取りたい」
- 「AI機能を追加してほしい」→ 本当の欲求：「作業時間を短縮したい」

### 優先度付けの基本ロジック
1. ユーザーが今一番困っていることを数字で確認（アンケート%）
2. 根本問題と表面的な要求を分離する
3. 「土台が安定してから機能を追加する」という順番を守る

### 対立意見への対応パターン
```
1. 相手の意見を肯定する（「おっしゃる通り〜」）
2. しかし、と続ける
3. 数字と論理で反論する
4. 「今はやらない」ではなく「次フェーズで必ずやる」と約束する
```

## PMが言ってはいけない表現パターン

| NG表現 | 理由 | 良い表現 |
|--------|------|---------|
| 「詳細を教えてください」 | 曖昧すぎて相手が答えられない | 「具体的にどんな場面でつまずいていますか？」 |
| 「〜ですか？」で終わる整理 | 確認質問になる | 「〜ということですね」と断言して整理 |
| 「まず優先順位を付ける必要があります」 | 当たり前すぎて根拠がない | 「67%が困っている〜を最優先にします」 |
| 「〜という形になります」 | 受け身・他人事感 | 「〜をやります」と能動的に宣言 |
